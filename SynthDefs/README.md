# SynthDefs

Put compiled SuperCollider `.scsyndef` files in this folder.

This folder now also contains:

- `catalog.json` for synthdef metadata and parameter contracts
- `manifest.tsv` for the repo build pipeline
- starter `.scd` source templates for the demo render/FX definitions

Current app behavior:

- on startup, the DAW looks for a `SynthDefs` directory
- if `manifest.tsv` and `sclang` are usable from the app runtime, it can try to auto-compile missing or stale `.scd` sources into `.scsyndef`
- if `.scsyndef` files are found, it sends `/d_loadDir` to `scsynth`
- track and insert `synthDefName` values will use those loaded definitions when the names match
- the DAW also reads `catalog.json` to attach descriptions and parameter metadata to known synthdef names
- if a requested synthdef is not available, the app falls back to SuperCollider's built-in `default` synth

Current starter definitions:

- `nebulaScene`
- `tapeBloom`
- `freezeFx`

Recommended naming:

- render tracks: use the `renderScript.synthDefName` field
- SC FX inserts: use the insert `superCollider.synthDefName` field
- MIDI generators currently still use control-bus mirroring rather than synthdef-driven event generation

If the app runtime cannot launch `sclang`, `.scsyndef` files still need to be compiled elsewhere or generated by another toolchain. The included `.scd` files are source templates plus an app-driven compile target when the local SuperCollider launch path cooperates.

Build script:

- validate only: `./scripts/build_synthdefs.sh validate`
- environment report: `./scripts/build_synthdefs.sh report`
- compile on a machine with working `sclang`: `./scripts/build_synthdefs.sh compile`

In-app workflow:

- open the DAW and use `Rebuild SynthDefs` in the SuperCollider routing panel
- the panel will then show source count, loaded synthdefs, auto-built synthdefs, and the active `SynthDefs` directory
